# VoiceBillRecord — 环境变量与密钥（安全说明）

本文档替代「通用 .env 说明」中与本项目无关的条目，**只描述本仓库（语音记账 + Neon + Serverless + 大模型）** 所需变量与安全边界。通用背景可参考仓库内 [App .env 模板与使用说明（示例 & 最佳实践）.md](../App%20.env%20模板与使用说明（示例%20&%20最佳实践）.md)。

## 1. 原则（比「能跑」更重要）

| 类别 | 变量示例 | 谁能接触 |
|------|----------|----------|
| **绝密（仅服务端进程）** | `DATABASE_URL`、`DEEPSEEK_API_KEY` | 仅 Vercel/本地运行 API 的环境；**永不**进入浏览器 bundle、**永不**进公开仓库 |
| **敏感（仍仅服务端）** | 将来若有 `JWT_SECRET`、Webhook 签名密钥 | 同上 |
| **可公开（仍非密钥）** | 前端自己的站点 URL（若需写死） | 可进静态资源，但不要与密钥混在同一文件 |

与 [需求文档.md](../需求文档.md)「八、九」及 [AGENTS.md](../AGENTS.md)「禁区」一致：**浏览器不持有数据库连接串、不持有大模型主 Key。**

## 2. 仓库内应提交什么

- **应提交**：根目录 [`.env.example`](../.env.example)（全部为占位符，无真实密钥）。
- **禁止提交**：`.env`、`.env.local`、含真实值的任何副本（已由根目录 `.gitignore` 忽略）。

复制模板：

```bash
cp .env.example .env
# 编辑 .env 填入 Neon 与 DeepSeek 等真实值；切勿 git add .env
```

## 3. 变量说明（与 `.env.example` 对齐）

- **`DATABASE_URL`**：Neon 控制台提供的 Postgres URL。**仅** Serverless（如 `api/expenses`）在运行时 `process.env.DATABASE_URL` 读取。  
  - 轮换：在 Neon 重置密码或新建 role 后更新 Vercel/本地 `.env`。
- **`DEEPSEEK_API_URL` / `DEEPSEEK_API_KEY` / `DEEPSEEK_MODEL`（可选）**：仅 **`/api/parse`** 调用 [DeepSeek API](https://api-docs.deepseek.com/) 时使用（兼容 OpenAI Chat Completions 形态）。前端只向**同源或你配置的 API 域名** `/api/parse` 发**用户已转写的文本**，不携带 DeepSeek Key。
- **`CORS_ORIGIN`**：允许携带 Cookie / 浏览器跨域访问 API 时的 `Origin` 白名单。静态与 API **同域部署**时，CORS 问题最少；若 Pages + 异域 API，必须在 API 侧严格白名单，**禁止** `*` 搭配凭据。分步操作见 [Vercel部署与CORS.md](./Vercel部署与CORS.md)。
- **`API_GATE_TOKEN`（可选）**：个人项目用来给 `POST /api/expenses` 加一道 `Bearer`。这是**弱门禁**，一旦写进前端仍可能被拷贝；仅作「防误扫」而非强身份。更强方案见需求文档「十三」扩展项。
- **`NODE_ENV`**：`development` / `production`，便于日志级别与断言（勿把密钥写进日志）。

## 4. 本地开发

1. 复制 `.env.example` → `.env` 并填值。  
2. 使用 `vercel dev` 或你选择的本地 Serverless 运行方式，确保**只有** Node 进程加载 `dotenv` / 平台注入（不要把整份 `.env` 传给前端 dev server 的 `define`）。

**不要**在前端 Vite/Webpack `define` 里注入 `DATABASE_URL` 或 `DEEPSEEK_API_KEY`。

## 5. 生产（以 Vercel 为例）

在 Project → Settings → Environment Variables 中为 **Production / Preview** 分别配置同名变量；值**只在控制台粘贴**，不要写进 `vercel.json` 或仓库 YAML。

- Preview 环境可使用**另一套** Neon branch 或同一项目的只读/开发库，避免误写生产数据。

## 6. CI（GitHub Actions）

- 当前仓库的 `ci` workflow **不**需要数据库或 DeepSeek 密钥即可跑 Vitest。  
- 若日后增加集成测试：在仓库 **Settings → Secrets and variables → Actions** 中创建 `DATABASE_URL` 等，在 workflow 里用 `secrets.DATABASE_URL` 注入；**禁止**在 YAML 里明文写连接串。

## 7. 泄露或误提交后的处理

1. **立刻轮换**：Neon 改密码、DeepSeek Key 在控制台作废并新建。  
2. **从 Git 历史中清除**敏感提交（需要 `git filter-repo` 或平台支持；仍假定密钥已泄露，以轮换为主）。  
3. 检查 Neon 审计日志是否有异常连接。

## 8. 与需求文档的对应关系

- 密钥不进前端、Neon 仅 Serverless：需求文档「八」「九」。  
- 可选前端 `API_GATE_TOKEN` 类弱门禁：[AGENTS.md](../AGENTS.md) 禁区第二条的补充说明。

## 9. 进阶（可选）

- 在 API 入口用 **schema 校验**（如 `zod` / `envalid`）对 `process.env` 做必填检查，缺变量时进程拒绝启动，避免「半配置上线」。  
- 对日志做脱敏：打印错误栈前剔除连接串片段。
